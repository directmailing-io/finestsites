Staging & Backups

Erklärt wie für einen Unternehmer — ohne Entwickler-Kauderwelsch

Kapitel 1

Backups — Was ist das und wer macht das?

Ein Backup ist eine gespeicherte Kopie deiner gesamten Datenbank. Wenn etwas schiefläuft, spielst du die Kopie zurück — so als hättest du auf "Rückgängig" gedrückt.

🎯 Kurze Antwort: Coolify hilft, reicht aber nicht allein

Coolify kann Backups machen — aber nur wenn es für die Datenbank zuständig ist, die innerhalb von Coolify läuft. Dein Setup ist anders: Die Datenbank läuft direkt auf dem VPS (nicht als Coolify-Container). Deshalb richten wir den Backup-Job direkt auf dem Server ein.

Das Gute: Der Backup-Job läuft danach vollautomatisch. Du musst nichts tun.

📦 Was wird gesichert?

Die Datenbank — also alle Nutzerdaten, Templates, Websites, Zahlungsinformationen, Submissions. Alles was in PostgreSQL gespeichert ist. Code-Dateien brauchen kein extra Backup, weil die in GitHub gesichert sind.

⚠️ Was passiert ohne Backup?

Wenn die Datenbank beschädigt wird oder der Server crasht, sind alle Nutzerdaten weg. Keine Backups = kein Netz unter dem Seil. Bei 100+ zahlenden Kunden ist das kein akzeptables Risiko.

Kapitel 2

Wie der Backup-Job funktioniert

Ein automatischer Job (Cron-Job) läuft täglich um 3:00 Uhr nachts, erstellt eine Sicherungskopie und legt sie auf dem Server ab. Ältere Kopien werden nach 14 Tagen automatisch gelöscht.

Täglich um 03:00 Uhr startet der Job

Vollautomatisch, ohne dass du etwas tun musst. Du schläfst, der Server arbeitet.

Automatisch

Die Datenbank wird als komprimierte Datei gespeichert

Zum Beispiel: backup-2026-07-12.sql.gz — etwa 1–5 MB pro Tag. Alle Daten drin: Nutzer, Sites, Zahlungen, alles.

Automatisch

Nach 14 Tagen werden alte Backups gelöscht

Der Server räumt sich selbst auf. 14 Tage bedeuten: Du hast immer 2 Wochen Zeit, einen Fehler zu bemerken und zurückzurollen.

Automatisch

Im Notfall: Backup einspielen

Du schreibst mir: "Ich brauche den Stand von gestern zurück." Ich erledige das in 10–15 Minuten.

Kapitel 3

Staging — Was ist das überhaupt?

Stell dir vor, du hast ein Restaurant. Bevor ein neues Gericht auf die Speisekarte kommt, testest du es in der Küche. Die Küche ist dein Staging. Die Speisekarte ist deine Live-App.

🎭 Eine einfache Erklärung

Production (live) ist die App, die deine Kunden gerade benutzen. Bei app.finestsites.io. Wenn hier etwas kaputtgeht, merken es alle sofort.

Staging ist eine identische Kopie davon — aber geheim, nur für dich und Claude. Hier werden Änderungen zuerst ausprobiert. Wenn sie funktionieren, kommen sie rüber zu Production.

🔄 Der typische Ablauf

Du beschreibst eine Änderung

"Ich will ein neues Template hinzufügen" oder "Der Anmeldebutton soll grün werden"

Claude baut die Änderung

Code wird geschrieben und auf Staging deployed. Noch sehen das nur wir.

Du testest auf Staging

Du gehst auf staging.app.finestsites.io und schaust: Sieht das gut aus? Funktioniert alles?

"Ja, passt" → Live schalten

Mit einem Befehl kommt die gleiche Änderung auf Production. Kein Experiment, kein Risiko — du hast es ja schon gesehen.

✅ Warum das wichtig ist

Ohne Staging: Du nimmst eine Änderung vor, und wenn etwas schiefläuft, bemerken es deine 100+ Kunden. Mit Staging: Fehler passieren nur auf dem Testsystem, das niemand außer dir sieht. Das ist der Unterschied zwischen einem Profi-Betrieb und einem Hobbyprojekt.
