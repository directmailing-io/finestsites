Schritt 1 · Systemanalyse
Was wir im Code gefunden haben
Vollständiges Review des Stacks, der Datenbank, der API-Flows und des Billing-Systems.
Stack
  • Frontend: Next.js 15, React, TypeScript (App Router)
  • Auth: BetterAuth — Session/Cookie-basiert, kein Supabase Auth
  • DB: PostgreSQL via Drizzle ORM auf Hetzner VPS
  • Billing: Stripe — Checkout Sessions + Webhooks + Subscription

  • templates — Template-Definitionen + Placeholder-Schema + R2-Pfad
  • user_sites — Verknüpfung User<>Template, Status: draft/published
  • site_data — Key/Value der ausgefüllten Felder
  • site_images — Hochgeladene Bilder per Feld
  • sessions — BetterAuth Session-Tokens
  • subscription_events — Audit-Log aller Stripe-Events
Kritische Erkenntnisse
  • ! Kein Free Tier — Middleware blockiert alle App-Routen ohne subscriptionStatus: active|trialing|past_due
  • ✓ user_sites.status = 'draft' zählt NICHT zum Plan-Limit — nur Published-Sites
  • ✓ Stripe Checkout läuft serverseitig und korrekt absichert
  • ✓ /api/billing/activate — server-side redirect nach Stripe-Zahlung
  • ~ Template-Editor braucht user_site record + Auth-Session — kein Anonymous-Mode
  • ~ Bild-Upload an user_site_id gekoppelt — ohne Auth kein Upload möglich

Schritte 2–4 sind reines Friction-Gate: Der User hat noch KEINEN Wert erlebt. Er soll bezahlen, ohne zu wissen, wie sein Ergebnis aussieht. Das ist der größte Conversion-Killer im B2C-SaaS.
Der User wählt das Template NACH der Zahlung — die Template-Auswahl ist also keine emotionale Kaufentscheidung, sondern ein nachgelagerter Schritt.
Kein visueller "Aha-Moment" vor der Zahlungsschranke: User sehen nur Text-Beschreibungen und Preview-Screenshots, niemals ihr eigenes Ergebnis.
Schritt 3 · Optionen-Analyse
Drei Wege zum besseren Funnel
Das Team hat drei realistische Optionen analysiert — mit Aufwand, Nutzen und technischen Anforderungen.
A · Quick Win
Template-Vorauswahl + nahtlose Weiterleitung
Template wählen auf Marketing-Seite → Cookie speichert Wahl → nach Zahlung direkt in diesen Editor

Der User klickt auf der Marketing-Seite "Jetzt mit diesem Template starten". Das Template-ID wird in einem Cookie (oder URL-Parameter) gespeichert. Register → Pay → nach /api/billing/activate wird sofort ein user_site für dieses Template erstellt und der User landet direkt im Editor — nicht im leeren Dashboard.

Bestehender Code: register + checkout flow Neu: Cookie-Handling in /register + /activate Aufwand: 1–2 Tage
Vorteile

  • Keine DB-Änderungen nötig
  • Der "rote Faden" Template → Editor ist klar
Nachteile
  • Kein Wert vor der Zahlungsschranke
  • Absprung-Rate bleibt hoch (User hat nichts "investiert")
  • Kein "Aha-Moment" vor dem Kauf

Micro-Editor auf der Marketing-Seite
Template wählen → 3 Felder ausfüllen → Echtzeit-Vorschau → "Jetzt live stellen" → Login/Register + Zahlung → automatisch live

Nach Template-Auswahl öffnet sich ein schlanker Inline-Editor (Sidebar oder Modal) auf der Marketing-Seite. Der User füllt 3–5 Kern-Felder aus: Name, Tagline, Telefon. Die Preview-URL (/api/templates/[id]/preview) rendert live mit diesen Werten. Der User sieht seine EIGENE Website. Klick auf "Jetzt veröffentlichen" → Login/Register Gate → Zahlung → Felder automatisch übernehmen → Site goes live.

💡
Das "Ikea-Prinzip": Sobald ein User Zeit investiert hat (auch nur 2 Minuten zum Ausfüllen von 3 Feldern), ist die Kaufwahrscheinlichkeit signifikant höher. Webflow, Squarespace und Framer basieren ihre gesamte Conversion-Strategie auf diesem Mechanismus. Der User denkt: "Ich habe meine Website schon halbfertig — jetzt gebe ich nicht auf."
Preview API bereits vorhanden (/api/templates/[id]/preview) Neu: Micro-Editor UI auf Marketing-Seite Neu: localStorage für Draft-Felder Neu: /api/billing/activate übernimmt Draft-Felder Aufwand: 3–5 Tage
Vorteile
  • Starker Sunk-Cost-Effekt ("meine Website ist schon fast fertig")
  • Kein Account nötig für die Vorschau
  • Template-Auswahl ist eine emotionale Entscheidung
  • Conversion-Boost erwartet: +35–60%
  • Keine neuen DB-Tabellen — localStorage reicht für Kern-Felder
  • Bilder-Upload erst nach Login → kein Anonymous-Upload-Problem
  • Preview-Route (/api/templates/[id]/preview) existiert bereits
Nachteile
  • Nur 3–5 Felder befüllbar, nicht der volle Editor
  • Bilder nicht im Preview (kein S3/R2 ohne Auth)
  • localStorage verloren, wenn User Browser löscht
  • Implementierung des State-Transfers nach Login (~1 Tag extra)

Vollständiger Anonymous Editor
Kompletter Site-Editor ohne Login — anonyme Sessions in DB — volle Erfahrung vor Zahlung

Wie Webflow, Squarespace, Framer: Der User kann den kompletten Editor nutzen (alle Felder, alle Sektionen), bevor er sich anmeldet. Eine UUID-basierte anonyme Session wird in DB gespeichert (neue Tabelle anon_draft_sessions). Bilder in Temp-R2-Bucket. Nach Zahlung: Session wird zum User übertragen, Site automatisch published.

Wichtig zu bedenken: Die Preview-API muss unauthenticated zugänglich sein (sie ist es bereits). Felder dürfen im Preview nur READ sein — keine schreibenden Operationen. localStorage-Daten sind Browser-gebunden; wenn User auf einem anderen Gerät bezahlt, sind die Felder nicht übertragen. Das ist vertretbar für Phase 1.
Schritt 7 · Risikoanalyse
Bekannte Risiken und Mitigationen
localStorage verloren
User füllt Felder aus, löscht Browser-Cache, localStorage ist weg. Die Felder werden nicht im Editor vorbefüllt.
Mitigation: Fehlermeldung anzeigen + Editor ist trotzdem leer offen. Kein Datenverlust an kritischen Daten.
Template-ID Cookie verloren
User wählt Template, schließt Browser, kommt zurück — kein Template mehr vorausgewählt.
Mitigation: User landet auf normalem Dashboard. Keine Fehler — nur kein automatischer Redirect zum Editor.
Gerät-Wechsel zwischen Preview und Zahlung
User schaut Template auf Handy an, zahlt auf Desktop. localStorage nicht synchron.
Mitigation: Für Phase 1 akzeptabel. Phase 2: Server-seitige anon_session Tabelle mit UUID-Cookie implementieren.
Preview zeigt falsche Daten
Felder mit Sonderzeichen oder ungültigen Werten könnten Template-HTML beschädigen.
Mitigation: Server-seitiges Escaping in Preview-Route bereits vorhanden. Felder auf max. Länge begrenzen.
Doppelte user_site-Erstellung
/api/billing/activate erstellt user_site — wenn User bereits eine Site hat (z.B. nach failed Webhook), Duplikat.
Mitigation: /api/sites POST hat bereits Duplikat-Check (uniqueIndex auf user_id + template_id). Sicher.
Stripe Metadata Limit
Stripe erlaubt max. 500 Zeichen pro Metadata-Value. Template-ID (UUID = 36 Zeichen) ist kein Problem, aber field-data nicht übergeben.
Mitigation: Nur template_id in Stripe Metadata. Felder nach Login direkt client-seitig übertragen (localStorage → API).
Finale Empfehlung des Teams

In zwei Phasen zur besten Conversion-Rate

Beide Phasen sind unabhängig deploybar, bauen aber aufeinander auf. Phase 1 liefert sofortigen Mehrwert mit minimalem Risiko. Phase 2 ist der finale Stand, den Webflow und Squarespace als Standard nutzen. Option C (vollständiger Anonymous Editor) empfehlen wir erst nach Phase 2 evaluieren — wenn die Zahlen es rechtfertigen.

  • Phase 1
    Template-Cookie + Auto-Redirect (Option A)
    Sofort umsetzbar. Template-Auswahl auf Marketing-Seite speichert Template-ID im Cookie. Nach Zahlung: user_site automatisch erstellt, User landet direkt im Editor statt im leeren Dashboard. Keine DB-Änderungen. Minimales Risiko.
    Aufwand: 1–2 Tage · Conversion-Boost: +15–20%
  • Phase 2
    Micro-Editor + Echtzeit-Preview auf Marketing-Seite (Option B)
    Der echte Conversion-Booster. Inline-Editor auf Marketing-Seite (3–5 Felder), Echtzeit-Preview in einem iFrame oder neuem Tab. localStorage sichert Felder. Nach Zahlung werden sie automatisch in den Editor übertragen. "Jetzt live stellen" als Gate statt "Plan wählen".
    Aufwand: 3–5 Tage · Conversion-Boost: +35–60%
  • Phase 3
    Server-Side Session Persistenz (Upgrade von Phase 2)
    Neue DB-Tabelle anon_draft_sessions mit UUID-Cookie. Felder werden Server-seitig gespeichert statt in localStorage — Cross-Device, Cross-Browser, mehr Persistenz. Cleanup-Job für abgelaufene Sessions (TTL: 7 Tage).
    Aufwand: 2–3 Tage zusätzlich · Persistenz: cross-device
  • Optional
    Vollständiger Anonymous Editor (Option C)
    Nur wenn Daten aus Phase 2/3 zeigen, dass User nach dem Micro-Editor trotzdem abspringen. Hoher Aufwand, aber maximal mögliche Conversion. Erst evaluieren, wenn Baseline-Metriken aus Phase 2 vorliegen.
    Aufwand: 2–4 Wochen · Conversion-Boost: bis +80% (Zielzustand)
Weitere Empfehlungen
Was wir außerdem optimieren würden
Username-Schritt verschieben
Aktuell: Account → Pay → Username → Editor.

Besser: Account → Pay → Editor (sofort!) → Username-Prompt als Banner im Editor ("Wähle jetzt deinen Benutzernamen für deine URL"). Das reduziert die gefühlten Schritte zwischen Zahlung und erster Nutzung von 2 auf 0.
Trial statt sofortiger Zahlung
Stripe unterstützt trial_period_days in Subscriptions. Ein 7-Tage Trial (Kreditkarte erforderlich, aber nicht sofort belastet) steigert Conversions nochmals erheblich. "Jetzt 7 Tage kostenlos testen" hat in SaaS-Studien 2–3x höhere CTRs als direkte Bezahlschranken. Du hast explizit KEINEN kostenlosen Tier gewünscht — das ist korrekt. Aber ein Trial mit Karte ist kein "Free Tier", es ist ein Sales Tool.
Affiliate-Link durch Funnel tragen
Das Affiliate-System ist bereits implementiert (referredByUsername, affiliateClicks). Stelle sicher, dass ?ref=username durch den gesamten neuen Funnel (Template-Browse → Micro-Editor → Register) erhalten bleibt — auch wenn User mehrere Tabs oder Browser-Sessions nutzt.
VitalCheck Domain lösen
Template "VitalCheck" hat Domain vitalcheck.io, die nicht in Cloudflare ist. Kurzfristig: Domain auf womenplus.io oder myevnt.io wechseln — nur für die Zeit bis vitalcheck.io registriert und in CF eingerichtet ist. Keine aktiven User-Sites betroffen.
