Deploy-Workflow

Wie Features sicher von der Idee zum Live-System kommen — mit 100+ zahlenden Kunden

🚨Die eine Regel die alles bestimmt

Jede Änderung geht zuerst auf Staging. Dann testest du. Dann sagst du "freigegeben". Erst dann kommt es auf Production.

Das gilt für Code-Änderungen, Preisänderungen, Datenbankänderungen, Konfigurationsänderungen — ausnahmslos.

Stripe Sandbox einrichten einmalig, dauert 5 Minuten

💡Sandbox = Test-Modus

Stripe hat 2024 "Test-Modus" in "Sandbox" umbenannt — es ist dasselbe. Im Sandbox-Modus passiert kein echtes Geld, du kannst Checkouts beliebig oft testen.

1

Stripe Dashboard öffnen

Geh zu dashboard.stripe.com → oben links auf "Sandbox" klicken (Toggle)

2

API-Keys kopieren

Entwickler → API-Schlüssel → Geheimen Schlüssel anzeigen → beide Keys kopieren (pk_test_... und sk_test_...)

3

Claude die Keys geben

Schreib mir die zwei Keys — ich trage sie in die Staging-Konfiguration ein und starte Staging neu. Dauert 2 Minuten.

4

Webhook für Staging registrieren

Stripe Sandbox → Entwickler → Webhooks → Endpunkt hinzufügen: https://staging.app.finestsites.io/api/webhooks/stripe → alle Events → Signing Secret kopieren → mir geben

Preisänderungen — der sichere Weg

💶Warum Preise besonders heikel sind

Preise in Stripe können nicht geändert werden, nur neue angelegt. Jeder Preis bekommt eine einmalige ID. Staging und Production brauchen deshalb eigene Preise im jeweiligen Stripe-Modus.

1

Stripe Sandbox: neuen Preis anlegen

Sandbox-Modus → Produkte → Preis hinzufügen → price_test_... ID notieren

2

Staging-Env updaten + testen

Claude trägt die Test-Price-ID in Staging ein → du testest den Checkout → Kreditkarte 4242 4242 4242 4242 funktioniert im Sandbox-Modus immer

3

Stripe Live: gleichen Preis anlegen

Live-Modus → gleiche Einstellungen → price_... ID notieren

4

Production-Env updaten + deployen

Nach deiner Freigabe → Live-Price-ID in Production → deployen

Do's und Don'ts

Richtig

  • Immer auf Staging deployen zuerst
  • Selbst auf Staging testen bevor Freigabe
  • Stripe Sandbox für alle Payment-Tests
  • DB-Migrations zuerst auf finestsites_staging
  • Explizit "freigegeben" sagen für Production
  • Stripe Restricted Key für Admin-Lesezugriff

Falsch

  • Direkt auf Production deployen
  • Live-Stripe-Keys in Staging eintragen
  • Production-DB für Tests nutzen
  • Migrations direkt auf Production laufen lassen
  • Annahmen treffen ohne zu testen
  • Staging überspringen weil "nur ein kleiner Fix"

Wenn Claude das nicht tut oder direkt auf Production gehen will — sage "Stop, erst Staging" und ich korrigiere mich sofort.
