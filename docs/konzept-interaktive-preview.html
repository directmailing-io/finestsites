Feature Konzept · v1.0

Interaktive Template-Vorschau
auf der Marketing-Seite.

Besucher der öffentlichen Vorlage-Seite sehen die echte Seite — mit vorausgefüllten Dummy-Daten. Sie können Farbthema, Sektionen und Bilder live anpassen und sofort sehen, wie ihre fertige Seite aussehen wird. Kein Account nötig.

Geschätzter Aufwand: 3–4 Tage
Erwartete Conversion: +40%
Kein Reload, vollständig live
Problem & Chance

Warum das ein Game-Changer ist.

Aktuell sieht ein Besucher auf /vorlagen/[id] statische Screenshots. Das ist schwach — er kann sich nicht vorstellen, wie SEINE Seite aussehen würde.

IST-Zustand
Statische Feature-Screenshots zeigen eine fremde Seite. Der Besucher muss sich die fertige Seite im Kopf vorstellen. Die Hürde ist groß: "Ich weiß nicht, ob das zu meiner Marke passt." Er registriert sich (noch) nicht.
SOLL-Zustand
Die echte Vorlage läuft als Live-Preview mit Dummy-Daten. Der Besucher wählt seine Farbe, schaltet Sektionen an/aus. Er sieht SOFORT seine Seite. Er klickt "Mit diesem Template starten" — und bringt seine Einstellungen mit.
💡
Key Insight vom UX-Team
Die Interaktion im Preview aktiviert den Endowment Effect: "Das ist MEINE Seite." Wer etwas konfiguriert hat, hört nicht mehr auf zu wollen. Die Hürde zur Registrierung sinkt drastisch.
User Journey

Wie der Besucher es erlebt.

6 Schritte von der ersten Sekunde bis zur Registrierung — mit den Einstellungen, die er bereits gemacht hat.

1
Besucher landet auf /vorlagen/[id]
Die Seite lädt wie bisher — Hero, Feature-Sektionen, "Weitere Templates". Neu: Darunter erscheint die Live-Preview-Sektion mit dem Headline "Wie deine Seite aussehen könnte."
UI — Neue Section auf der Marketing-Seite
2
iframe lädt mit Demo-Daten (keine Auth nötig)
Das iframe ruft /api/templates/[id]/public-preview?data=BASE64_DEMO_DATA auf. Die Seite zeigt die echte Vorlage mit vorausgefüllten Dummy-Werten (Vorname, Foto, Texte usw.).
API — Neue öffentliche Route (kein Auth) Neu: demo_data aus preview_config
3
Editor-Panel erscheint links neben dem iframe
Ein abgespeckter Editor zeigt nur die konfigurierten "editable_fields": Farbthema (card_select), Header-Bild (toggle/image), Sektionen (toggles). Identische UI-Komponenten wie im echten Editor.
UI — Simplified Editor Panel
4
Besucher interagiert — iframe aktualisiert live
Klick auf "Grün" → postMessage finestsites:updateField an iframe → Farbe wechselt sofort ohne Reload. Toggle Sektion → postMessage finestsites:toggleSection → Sektion fährt animiert aus. Alles mit dem bekannten Live-Engine.
Live: postMessage API (bereits vorhanden)
5
CTA: "Mit dieser Konfiguration starten"
Der Klick speichert die aktuellen Einstellungen als Base64-JSON im Cookie fs_preview_config und redirectet zu /register?template=[id]. Zusätzlich zum bestehenden template-Cookie.
Cookie: fs_preview_config (7 Tage) Vorkonfiguration wird mitgenommen
6
Nach Registrierung: Editor startet mit Preview-Einstellungen
Der Editor liest den Cookie und wendet Farbthema + Sektions-Einstellungen bereits an. Der User sieht sofort "seine" Seite — Aha-Moment. Cookie wird gecleared.
Cookie → DB → Clear Nahtlose Übergabe

Technische Architektur

API & Datenfluss.

Alle benötigten Bausteine sind bereits vorhanden. Wir brauchen eine neue öffentliche Route und eine neue Config-Spalte. Der Rest ist Verdrahtung.

Besucher (anonym)
──▶
finestsites.io/vorlagen/[id]
│ iframe src
· ──▶
GET /api/templates/[id]/public-preview?data=BASE64
NEU · kein Auth
│ liest
· ──▶
templates.preview_config
NEU · demo_data + editable_fields
│ liest HTML
· ──▶
R2 / template HTML
bereits vorhanden
Live-Interaktion
──▶ postMessage(finestsites:updateField | finestsites:toggleSection)
│ zu iframe
· ──▶
Live Runtime Engine (bereits in iframe)
bereits vorhanden · kein Reload
CTA-Klick
──▶ fs_preview_config Cookie setzen ──▶
/register?template=[id]
│ nach Login
· ──▶
Editor liest Cookie → wendet Einstellungen an → Clear
NEU
🔑
Kein Auth für Public Preview
Die neue Route /api/templates/[id]/public-preview läuft ohne Session. Sie bekommt Rate-Limiting (60 req/min per IP) und gibt nur Templates mit status='published' zurück. Die ?data= Base64 payload wird mit einem max. von 8KB limitiert.
Datenschema

preview_config — das neue Feld.

Eine neue JSONB-Spalte in der templates-Tabelle. Der Admin pflegt diese Daten im Admin-Panel. Einmal konfiguriert — für alle Besucher.

// templates.preview_config — Beispiel für "Das Optimalset" (womenplus)
{
  // Alle vorausgefüllten Demo-Werte — werden im iframe angezeigt
  "demo_data": {
    "vorname": "Anna",
    "nachname": "Müller",
    "profilbild": "https://demo.womenplus.io/assets/anna-demo.jpg",
    "header_bg": "https://demo.womenplus.io/assets/header-demo.jpg",
    "erfahrung_text": "Ich nutze das Optimalset seit 3 Monaten...",
    "theme_color": "gruen",
    // ... alle weiteren Felder aus dem placeholder_schema
  },

  // Welche Felder der Besucher im Editor-Panel ändern kann
  "editable_fields": [
    {
      "key": "theme_color",
      "label": "Farbthema",
      "type": "card_select", // kopiert aus placeholder_schema
      "preview_hint": "Wähle deine Akzentfarbe"
    },
    {
      "key": "header_bild",
      "label": "Header-Hintergrundbild",
      "type": "toggle" // zeigt/versteckt Demo-Bild
    }
  ],

  // Welche data-section Keys der Besucher an-/ausschalten kann
  "editable_sections": [
    { "key": "video", "label": "Erklärvideo", "default_on": true },
    { "key": "erfahrung", "label": "Erfahrungsbericht", "default_on": true },
    { "key": "vorteile", "label": "Die Vorteile", "default_on": true },
    { "key": "ueber_mich", "label": "Über mich", "default_on": false }
  ],

  // Optionale Vorkonfiguration die nach Registration in den Editor übertragen wird
  "carry_over_fields": ["theme_color", "editable_sections"]
}
⚠️
Sicherheit: demo_data muss sauber sein
Demo-Daten werden in der öffentlichen Preview gerendert. Sie dürfen kein XSS enthalten. Bei der Speicherung im Admin-Panel wird die preview_config durch denselben Sanitizer geleitet wie der Template-Engine-Input.
Feldtypen im Preview-Editor

Was kann der Besucher editieren?

Jeder Feldtyp aus dem placeholder_schema kann für die Preview-Konfiguration genutzt werden. Der Admin entscheidet pro Template, welche Felder sinnvoll sind.

Typ (aus Schema) UI im Preview-Editor Live-Update Methode Empfohlen für Aufwand
card_select Farbige Karten (wie im Editor) postMessage updateField → CSS-Variable Farbthema, Stil-Variante Niedrig
toggle On/Off Switch postMessage toggleSection Sektionen an/aus Niedrig
text Inline-Textfeld (ein Zeile) postMessage updateField → Live-Text Name, Stadt, Slogan Niedrig
image Demo-Bild Toggle / Beispiel-Galerie postMessage updateField → img src Header-Bild, Profilbild Mittel
dropdown Kleines Select-Menü postMessage updateField Branche, Region, Sprache Niedrig
textarea Mehrzeiliges Textfeld postMessage updateField → Live-Text Kurzbeschreibung, Slogan Mittel
loop Nicht empfohlen für Preview — Zu komplex für Erstbesucher Nicht Phase 1
✅
Design-Entscheidung: Maximal 4–5 editierbare Optionen
Wir empfehlen pro Template maximal 4–5 editierbare Controls. Mehr überfordert den Besucher. Das Ziel ist: schnell verstehen, schnell begeistert sein, schnell registrieren. Nicht: maximale Flexibilität.
Admin-Workflow

So konfiguriert man ein neues Template.

Einmal einrichten — dauerhaft für alle Besucher aktiv. Der Admin-Workflow für Template-Autoren.

Schritt 1 — Demo-Daten definieren
Im Admin-Panel bei Templates: Preview-Konfiguration öffnen. Dort alle Felder aus dem placeholder_schema mit Dummy-Werten befüllen. Das sind die Werte, die der Besucher als Startpunkt sieht. Tipp: Echte, ansprechende Demo-Daten nehmen (wie auf demo.womenplus.io).
vorname = "Anna" profilbild = demo-url theme_color = "gruen" erfahrung_text = "..."
Schritt 2 — Editierbare Felder wählen
Aus dem placeholder_schema auswählen, welche 3–5 Felder der Besucher ändern kann. Empfehlung: immer Farbthema + 2–3 Sektions-Toggles. Optional: Textfeld für den Vornamen (macht es persönlich).
Schritt 3 — Editierbare Sektionen definieren
Aus den data-section Keys im Template HTML auswählen, welche Sektionen der Besucher an-/ausschalten kann. Default-Zustand (on/off) festlegen.
Schritt 4 — Vorschau testen und veröffentlichen
Im Admin-Panel gibt es einen "Preview testen" Button, der die öffentliche Vorschau im neuen Tab öffnet. Nach Zufriedenheit: Status auf "published" → die interaktive Preview erscheint auf der Marketing-Seite.
Implementierung

Roadmap in 3 Phasen.

Schritt für Schritt vorgehen. Phase 1 ist MVP und bringt bereits messbaren Mehrwert.

Phase 1 · ~2 Tage
MVP: Live-Preview mit Color-Picker & Sektions-Toggles
Der schnellste Weg zum Mehrwert. Alles was funktionieren muss, damit ein Besucher die Seite als "seine" sieht.
DB Migration: preview_config JSONB Spalte zu templates hinzufügen
Neue API Route: GET /api/templates/[id]/public-preview (kein Auth, Rate-Limit)
Neue Section auf /vorlagen/[id] Page: Split-Panel mit iframe + Editor-Panel
Editor-Panel: card_select (Farb-Cards) + toggle (Sektionen) Komponenten
postMessage Bridge: Editor → iframe (bereits vorhanden, nur verdrahten)
CTA-Button: Einstellungen in Cookie speichern + redirect zu /register
Admin-Panel: Neues "Preview-Konfiguration" Tab bei Template-Edit
Das Optimalset (womenplus) als erstes Template konfigurieren
Phase 2 · ~1 Tag
Cookie-Übernahme in den Editor nach Registration
Der "Aha-Moment" nach dem Login: "Das ist schon genau meine Seite."
Editor beim Laden: fs_preview_config Cookie lesen
Konfigurierte Felder direkt in site_data schreiben
Sektions-Toggles aus Cookie auf den initialen Preview-State anwenden
Cookie clearen nach Anwendung
Toast/Banner: "Wir haben deine Vorschau-Einstellungen übernommen ✓"
Phase 3 · ~1 Tag (optional)
Erweiterte Personalisierung & Analytics
Mehr Tiefe, mehr Conversion-Insights.
Text-Feld: Vorname eingeben → live im iframe anzeigen
Bild-Varianten: Vordefinierte Demo-Bilder auswählen (kein Upload)
Analytics: Wie lange interagiert? Welche Farbe gewählt? → Conversion-Rate per Theme
Mobile-optimiertes Layout: Tab "Editor" ↔ "Vorschau" statt Split-Screen
Share-Link: Konfiguration als URL-Hash teilen (z.B. für Social-Media-Posts)
