The Core Problem: Zero Value Before Commitment

The current funnel asks users to register before they see a single pixel of what their website could look like. This means there is no "aha moment" before the commitment wall — resulting in high drop-off at registration and low activation rates downstream.

Current (Painful) Flow
Land on Marketing Page
→
Register (email + password + username)
→
Pick a Plan
→
Pick a Template
→
Fill Placeholders
→
Publish

User sees NO value before registering. The "why should I care?" question is never answered before the friction wall hits.

Option A
Option A

Template Cookie

Minimal friction reduction — template pre-selection via cookie

1–2 days

The user selects a template on the marketing page and that choice is stored in localStorage or a cookie. After registration, the template is automatically pre-selected and the user lands directly in the editor — skipping the template selection step.

Pros
  • Extremely fast to implement
  • Works immediately, no backend changes
  • No anonymous state management complexity
  • User made a template choice — some commitment
  • Skips one step post-registration
Cons
  • User still registers before seeing real value
  • Template "thumbnails" don't convey actual output
  • No meaningful "aha moment" before the wall
  • Lowest conversion impact of all options
  • Cookie may be lost (cross-device, private browsing)
💻
Senior Dev
Feasibility

Trivial implementation. Add a localStorage.setItem('templateId', id) on click and read it post-register. Risk: near zero. Maintenance: near zero. This is a safe starter but offers minimal competitive edge.

📈
Growth
Conversion Impact

Very low impact. The registration wall is still the primary drop-off point. Template pre-selection doesn't justify the effort — it only marginally reduces post-registration friction. Baseline improvement at best.

When a user clicks a template, they land on a dedicated preview page — not just a thumbnail. The preview renders the actual template HTML with realistic sample data (business name, phone, photos, etc.). This is the aha moment: the user sees exactly what their site will look like. After clicking "Use This Template", they register and land directly in the editor — with the template pre-loaded and template-selection step skipped entirely.

Key insight: The preview page must be built anyway for SEO, ad campaigns, and template discovery. A+ repurposes this necessary work as the primary conversion driver. Double ROI on the same implementation effort.

Sample data per template — this is one-time work. Each template needs a realistic JSON payload (business name, slogan, contact details, 1–2 placeholder images). Once created, it serves every user who previews that template forever. The admin panel can expose this as an editable field.

Pros
  • Strong value demonstration before registration
  • User sees EXACTLY what they're signing up for
  • State management is trivial (cookie/localStorage)
  • Preview page doubles as SEO/marketing asset
  • Natural, excited journey — register at peak enthusiasm
  • Can be linked from ads: "Preview the Fitness template"
  • Template selection step skipped post-register
Cons
  • Requires good sample data per template (one-time effort)
  • Registration wall still exists — just better positioned
  • Preview images may not match user's actual niche images
  • Need to handle preview page routing and SEO metadata
💻
Senior Dev
Implementation Notes

Route: /templates/[slug]/preview. Render template HTML server-side with sample data from DB. "Use This Template" button sets localStorage.setItem('selectedTemplate', slug) and redirects to /register. Post-register middleware reads the value and redirects to plan → editor. Risk: low. The template engine already exists — the preview route is 80% there.

📈
Growth
Conversion Psychology

This creates an Endowment Effect — once users see "their" site rendered live, they feel partial ownership. The psychological commitment at the "Use This Template" click is high. Registration at this point is completing a goal, not an arbitrary wall. Expect meaningful lift in registration-to-activation rates.

Option B — Recommended Phase 2
Option B ★ Phase 2 Pick

Pay First, Set Password Later

Eliminates registration friction entirely — email + plan + Stripe, account created post-payment

3–5 days
Proposed Flow
Land
→
Browse Templates
→
Live Preview (from A+)
→
Enter Email Only
→
Pick Plan + Stripe Checkout
→
Success: "Set Your Password"
→
Dashboard + Editor (template pre-loaded)

The biggest conversion killer is the question: "Why do I need to register before I know if I want this?" Option B eliminates this entirely. The user enters only their email address, picks a plan, and goes straight to Stripe Checkout. On payment success, a BetterAuth user account is created server-side (via webhook), and the user sets their password on the success page or via a "complete your account" email link.

Stripe handles identity. The email entered before checkout becomes the Stripe customer email. The webhook receives it, creates the BetterAuth account with a temporary password, and sends a "Welcome — please set your password" email via Resend.

Implementation path: New /api/billing/checkout-v2 endpoint accepting { email, templateId, planId } → creates Stripe Checkout Session with metadata → existing webhook creates BetterAuth user on checkout.session.completed → sends welcome email with set-password link. The templateId is stored in Stripe session metadata and read post-payment.
Pros
  • Eliminates the registration friction wall entirely
  • User commits at peak excitement (just saw live preview)
  • Highest conversion potential for a paid-first SaaS model
  • Stripe email = account email, zero duplication
  • templateId survives in Stripe metadata — reliable
  • Aligns with best practices (Lemon Squeezy, Paddle flows)
Cons
  • "Orphaned payments" — paid but never set password
  • More complex webhook logic required
  • Need "resend set-password email" flow
  • BetterAuth user creation from webhook needs testing
  • Edge case: user pays with different email than typed
💻
Senior Dev
Risk & Architecture Notes

The webhook is the critical path. BetterAuth's auth.api.signUpEmail or direct DB insert must be tested for idempotency — Stripe webhooks can fire multiple times. Store a stripeSessionId on the user row as a unique key. "Orphaned payments" should trigger an automated follow-up email after 24h. The existing Stripe webhook handler in /api/webhooks/stripe is the right extension point.

📈
Growth
Conversion Impact

This is the highest-leverage change in this document. Industry data: removing mandatory pre-payment registration typically lifts checkout completion by 20–35%. Users who've seen the live preview (A+) and clicked "Get Started" are already emotionally invested — the fewer clicks to payment, the better. "Set password later" is now standard in B2B SaaS (Linear, Loom, Notion invites).

A full anonymous session where users edit the entire site before registering. All state is persisted to localStorage or a temporary server-side session ID. On registration, the anonymous session is merged into the new user account.

Pros
  • Maximum "aha moment" — user completes their site first
  • Highest possible emotional investment before paywall
  • Sunk-cost drives conversion at registration
Cons
  • 2–4 weeks of engineering work
  • Image upload before registration is architecturally complex
  • Anonymous session expiry and cleanup logic needed
  • Session merge edge cases (email conflict, partial data)
  • Overkill given A+ + B already solve the core problem
  • Risk of "build it and they don't convert" — users edit but don't pay

C is not wrong — it's just disproportionate to the problem at this stage. Revisit after A+ and B are live and validated.

Team Perspectives

Three Angles on This Decision

💻
Senior Developer Perspective
Feasibility, Risk & Technical Architecture

A+ is the right first move. The template engine in src/lib/utils/template-engine.ts already handles placeholder substitution. Building a /templates/[slug]/preview route that feeds sample data through that engine is a 2-day task at most. The cookie/localStorage state is trivial — read it in the post-registration redirect middleware.

B requires careful webhook engineering. The existing /api/webhooks/stripe/route.ts is the right extension point. Critical: the webhook handler must be idempotent. Use stripeSessionId as a unique constraint on the users table to prevent duplicate account creation on webhook retries. BetterAuth user creation from a server-side context (not a request) needs testing — use the BetterAuth admin API or direct DB insert with proper password hashing.

Biggest technical risk in B is the "orphaned payment" scenario — user pays, webhook fires successfully, but the set-password email goes to spam or is never clicked. Mitigation: show the set-password form inline on the Stripe success page (same tab) AND send the email. Add a cron job (/api/cron/ already exists) to detect unactivated accounts after 24h and resend the email. After 7 days with no activation, flag for manual review and optionally refund automatically.

📈
SaaS Marketing / Growth Perspective
Conversion Psychology, User Journey & Best Practices

The "aha moment" is everything. In SaaS, conversion correlates with time-to-value. Every second a user spends filling forms before seeing the product output is a second they're asking themselves "is this worth it?" Option A+ creates the aha moment before the registration wall — the user sees their site (even with sample data) and thinks "that's exactly what I need." The registration at that point feels like completing a process, not starting one.

Option B is the long-term play. The best-converting SaaS funnels in 2025–2026 follow the pattern: see value → provide email → pay → set password. This mirrors e-commerce checkout flows that users are already trained on. Removing the "create an account with username and password" step from the pre-payment journey is consistently shown to lift paid conversion by 20–35% in B2C SaaS. The "set password later" moment, done right, feels like a reward — "your account is ready, just secure it."

Sequencing A+ before B is strategically sound. A+ creates the preview page infrastructure. B reuses it and adds the Stripe layer. This means B is only 3–5 days of additional work on top of A+, not 3–5 days from scratch. Plus, A+ gives you real data: which templates get the most preview clicks? That informs which template to pre-load in B's checkout flow.

Ad channel unlock: Preview pages (A+) are individually linkable. You can run ads targeting "fitness instructor website" → directly to the Fitness template preview page. This is not possible with the current flow. It's a significant top-of-funnel multiplier.

🌟
Apple UX / UI Perspective
Experience Design & Friction Reduction

The preview must feel like the real thing. If the preview page shows a watered-down or obviously "demo" version of the template, it undermines trust. Sample data must be aspirational but believable — a real business name, professional-sounding tagline, a high-quality stock image that matches the niche. "Sarah's Yoga Studio" with a beautiful hero photo converts better than "Lorem Ipsum Business Name."

The CTA on the preview page must be singular and clear. One button: "Use This Template — it's free to start." No pricing, no feature lists, no distractions. The goal of the preview page is one thing: get the user to click that button. Pricing can be shown on the plan selection page, after registration.

For Option B: The email-only entry step should be a minimal, full-screen centered form — not buried in a nav flow. "Enter your email to get started" with a "Continue" button. No password field on this screen. The Stripe checkout handles the rest. The success page should immediately show "Your site is ready — secure your account" with the password form inline, no redirect needed.
